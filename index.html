이렇게 커밋을 하면 하나의 스냅샷이 생성됩니다.

$ vim index.html

가르키지만 master는 기존 스냅샷을 가르킵니다.


새로운 브랜치를 하나 더 만들어 볼까요?
$git checkout -b 'hotfix'

$ vim index.html
$ git commit -a -m 'fixed the broken email address'

새로운 브랜치 hotfix가 생성되었습니다. 여러개의 브랜치들이 이런 모양으로 생성되고 연결됩니다.
물론 브랜치에서 브랜치가 만들어지죠,, 
엄연히 말하면 master도 브랜치이기때문입니다.
브랜치 hotfix와 master를 합쳐볼까요?
$ git checkout master
$ git merge hotfix
Updating f42c576..3a0874c
Fast forward

브랜치 hotfix의 스냅샷은 브랜치 master의 스냅샷보다 최신이므로 hotfix 스냅샷으로 master 브랜치 포인터를 가르킵니다.

$ git checkout master

어떤 원리로 Merge가 되는지 궁금하지 않나요?

hotfix와 master 브랜치의 Merge 방법 분석!

hotfix 브랜치가 가르키는 커밋의 부모가 master 브랜치가 가르키는 커밋과 일치하므로 master 브랜치가 가르키는 포인터가 C4를 가르키게 됩니다.
포인터가 앞으로 이동하는 방식이므로 
Fast Forward라고 합니다.
iss53과 master 브랜치간의 Merge 방법 분석!

iss53의 커밋의 부모가 앞에서와는 다르게 
master 커밋을 가르키지 않아요. 

이러면 Git은 각 브랜치가 가르키는 커밋 2개와 공통 조상 하나를 사용하여 3-way Merge를 합니다.
3-way Merge 결과를 별도의 커밋을 만들어 master 브랜치 포인터가 그 커밋을 가르키도록 이동합니다.
이러한 커밋은 부모가 여러개이며 Merge 커밋이라고 합니다.


혹시 Merge하면 충돌 날 수 있잖아요? 이 부분은 어떻게 할까요?

소스를 잘못 작성해서 커밋을 했을 때 수정하는 방법입니다.

    
git checkout --	// 수정했지만 아직 스테이지에 추가하지 않은 파일을 되돌릴 때,
    
git reset HEAD	// 수정했고 스테이지에 추가한 파일을 다시 스테이지에서제거 할 때,
    
git fetch origin	// 수정했고 로컬저장소에 commit 한 파일을 되돌릴 때,
